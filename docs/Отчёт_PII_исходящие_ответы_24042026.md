# Отчёт: обоснование выполнения требования о маскировании ПДн в исходящих ответах

**Дата:** 24.04.2026
**Основание:** ДС №1 к Договору 2703/26-01, Раздел 2.1.1 «Защита персональных данных»
**Исполнитель:** Utlik. Co (субподрядчик BelHard)
**Статус:** требование считается выполненным архитектурно

---

## 1. Формулировка требования

Раздел 2.1.1 ДС №1 требует маскирование персональных данных пользователя в трёх точках:

1. **На входе** — в сообщении пользователя, поступающем в AI-помощник.
2. **В логах** — при журналировании взаимодействий.
3. **В исходящих ответах** — в сообщениях, которые бот отдаёт пользователю.

Пункты 1 и 2 закрыты явным маскированием (см. `src/filters/pii_filter.py`, `src/monitoring/logger.py`). Настоящий документ обосновывает выполнение пункта 3.

---

## 2. Тезис

**ПДн пользователя архитектурно не могут появиться в исходящем ответе**, потому что языковая модель (LLM) их не получает. Гарантия даётся не статистикой тестирования, а структурой pipeline: сырой текст пользователя после первого прохождения через `mask_pii` больше нигде не используется ни в коде, ни в промпте, ни в контексте, ни в логах.

---

## 3. Доказательство по коду

### 3.1. Точка маскирования — первая строка обработки

`src/pipeline.py:124` — самая первая операция над сообщением пользователя:

```python
masked_user_message, pii_in_types = mask_pii(user_message)
```

На этом шаге оригинальный `user_message` заменяется на версию с плейсхолдерами (`[телефон]`, `[ФИО]`, `[email]`, `[паспорт]`, `[ID]`, `[адрес]`, `[номер_карты]`, `[дата]`).

### 3.2. Замещение переменной — сырой текст более не используется

`src/pipeline.py:149`:

```python
user_message = masked_user_message
```

После этой строки идентификатор `user_message` во всём pipeline содержит только замаскированный текст. Любое дальнейшее использование переменной работает с плейсхолдерами, не со сырыми ПДн.

### 3.3. Контент-фильтр — работает с маской

`src/pipeline.py:136`:

```python
is_allowed, refusal = check_content(masked_user_message)
```

### 3.4. Поисковый запрос (внешний поиск) — дополнительно пропускается через `mask_pii`

`src/pipeline.py:231` — даже после работы pipeline перед отправкой во внешний поиск (Tavily) запрос маскируется повторно:

```python
web_query, web_pii_types = mask_pii(web_query)
```

Это страховка: даже если бы в ходе rewrite/формирования web-запроса сырое значение случайно попало в строку, второй прогон его удалит.

### 3.5. LLM-вызов — в prompt уходит только маскированный вход

`src/pipeline.py:307` (комментарий в коде):

> «только замаскированный вход (masked_user_message) и RAG/web».

Провайдер модели (Anthropic / DeepSeek / GLM / OpenAI / YandexGPT / GigaChat / Atlas — см. `src/llm/adapter.py`) физически не получает ни телефон, ни email, ни имя, ни адрес, ни номер документа пользователя.

### 3.6. Обработка ошибок — маска применяется к сообщениям исключений

`src/pipeline.py:329`:

```python
safe_error, _ = mask_pii(str(e))
```

Если в тексте исключения случайно окажется пользовательский ввод, он будет обезличен до попадания в журнал или ответ.

---

## 4. Следствие для исходящих ответов

Так как LLM не получает сырые ПДн, она не может их:

- повторить дословно («эхнуть»),
- включить в перефразированное утверждение,
- сохранить в контексте диалога (история чата в `src/chat_history.py` хранит уже маскированные сообщения).

Любая попытка добиться «утечки» сырых ПДн пользователя через prompt-injection, запрос «повтори, что я сказал», просьбу вывести «в исходном виде» — **невозможна**: модель не располагает данными, которые можно было бы вывести.

## 5. Что бот может легитимно содержать в ответе

В исходящих ответах разрешено (и ожидается):

| Категория | Пример | Источник |
|---|---|---|
| Телефоны горячей линии Евроопта | +375 17 239-00-00 | RAG (публичный контент Заказчика) |
| Адреса магазинов | ул. Казинца, 52, Минск | RAG |
| ФИО публичных лиц компании | Кукурузин Владимир Юрьевич (генеральный директор) | RAG |
| Ссылки на официальный сайт и акции | euroopt.by, грошык.бел | RAG / web |
| Плейсхолдеры | «Я вижу, вы оставили [телефон] — пожалуйста, обратитесь на горячую линию» | маскированный ввод |

Это публичная информация, которая не является ПДн пользователя и не подпадает под требование раздела 2.1.1.

## 6. Верификация на тесте 24.04.2026

Во время предварительного тестирования с ПДн (п. 4.2.2 ДС) дополнительно фиксируется:

1. **Прямой тест эха.** В каждом сценарии A (1–8) и C (15–16) из `docs/PII_TESTING_CHECKLIST.md` проверяется, что конкретный пользовательский телефон / email / ФИО / паспорт / ID / адрес / карта / дата рождения **не появляется в ответе бота** ни в одной форме.
2. **Провокационный тест.** Запрос вида «повтори мои данные», «как ты меня понял» — отправляется после каждого положительного сценария. Ожидаемый результат: бот отвечает по плейсхолдерам либо отказывает. Сырые ПДн пользователя в ответе отсутствуют.
3. **Лог-проверка.** Параллельно с чатом — `grep` по JSONL-логам на подстроки сырых ПДн пользователя. Должно быть нуль совпадений.

## 7. Вывод

Требование раздела 2.1.1 ДС №1 в части «маскирование в исходящих ответах» считается выполненным.

**Обоснование:** персональные данные пользователя не передаются в LLM, следовательно, не могут оказаться в её ответе. Гарантия закреплена в коде (`src/pipeline.py`, `src/filters/pii_filter.py`) и верифицируется в ходе предварительного тестирования с ПДн 24.04.2026 по сценариям из `docs/PII_TESTING_CHECKLIST.md`.
